# AWS Infrastructure with Terraform

This repository stores Terraform files for all AWS resources managed by the infrastructure team.

## Basic Concepts

A _stack set_ is a directory containing Terraform files, in another word, it is a Terraform root module. We typically group stack sets by application teams, and all are stored under the direction `applications`. For example, if an application team named `helios` needs a stack set for a MariaDB, the stack set name can be `helios/mariadb`, and the directory holding the terraform files is `applications/helios/mariadb`.

A _stack_ is a stack set used for a given AWS account in an AWS region. Each stack has a Terraform state file,
stored in the S3 bucket with the name _stack-set_/_aws-acount_/_aws-region_/terraform.tfstate. Continuing with above MariaDB example, if we run the stack set for the AWS account 821080507765 in the `us-east-2` region, the S3 bucket file name will be `helios/mariadb/821080507765/us-east-2/terraform.tfstate`.

This repository is integrated with Github Actions. When a pull request is created, the Github Actions automatically triggers `terraform plan`, and puts the result as a comment to the pull request.  Once the pull request is reviewed (including reviewing the plan) and merged, the Github Actions will run `terraform apply` to create the infrastructure.

The file `github.env` in this directory is used to define environment variables specifying which stack set and for which AWS account and region. The following is a sample `github.env` which shows the exact environment variable names:
```
STACK_SET=app1/dynamodb
AWS_ACCOUNT=821080507765
AWS_REGION=us-east-2
```

## For Developers

The developers outside the infrastructure team can use this repository
for self service (authoring Terraforms, but still requires the infrastructure team to review and approve pull requests)

The following are the main steps

* Clone this repository
* Create or modify terraform in your directory `applications/`*application-name*`/`*stack_set*.
  Note you do not need to provide `terraform` and `provider` blocks - these will be automatically generated by Github Actions, also by the
  script `scripts/trplan` when you test locally
* Update the 3 environment variables in `github.env` as appropriate
* Create a pull request
* Wait for a minute and look for an automatically generated comment in the pull request that contains the output of `terraform plan`, and verify the planned changes.
* Request the Infrastructure team to review and merge the pull request 

## Miscellaneous Topics

### Tag a Version of the Current Version

To see the current tags with messages, use `git tag -n`

To create a new tags:

```
    git checkout main
    git tag v1.2.0 -m "Create application/stack org structure"
    git push origin --tags
    git checkout develop
```
### Destroy a Stack

After updating `github.env` appropriately, run the script `scripts\tfdestory`.

_We will intergrate with Github Actions in the future, probably by adding an environment variable `DESTROY=true` to `github.env`_.
