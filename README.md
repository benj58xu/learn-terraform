# AWS Infrastructure with Terraform

This repository stores Terraform files for all AWS resources managed by the infrastructure team.

## Basic Concepts

A _stack set_ is a directory containing Terraform files — essentially a Terraform root module. Stack sets are typically organized by application team and stored under the `applications` directory. For example, if an application team named `helios` needs a stack set for a MariaDB database, the stack set would be named `helios/mariadb`, with Terraform files stored in `applications/helios/mariadb`.

A _stack_ is a specific deployment of a stack set to a particular AWS account and region. Each stack has a Terraform state file stored in S3 with the path: _stack-set_/_aws-account_/_aws-region_/terraform.tfstate. Continuing with the MariaDB example, if you deploy the stack set to AWS account 821080507765 in the `us-east-2` region, the state file would be at `helios/mariadb/821080507765/us-east-2/terraform.tfstate`.

This repository is integrated with GitHub Actions. When you create a pull request, GitHub Actions automatically runs `terraform plan` and posts the results as a comment. After the pull request is reviewed and approved, GitHub Actions runs `terraform apply` to deploy the infrastructure.

The `github.env` file defines environment variables that specify which stack set to deploy and to which AWS account and region. Here's a sample showing the required variables:
```
STACK_SET=app1/dynamodb
AWS_ACCOUNT=821080507765
AWS_REGION=us-east-2
```

## For Developers

Developers outside the infrastructure team can use this repository for self-service (authoring Terraform files while the infrastructure team reviews and approves pull requests)

Follow these steps:

* Clone this repository
* Create or modify Terraform files in `applications/`*application-name*`/`*stack_set*
  * Note: You don't need to provide `terraform` and `provider` blocks—these are automatically generated by GitHub Actions and the `scripts/tfplan` script for local testing
* Update the three environment variables in `github.env` as needed
* Create a pull request
* After a minute, review the automatically generated comment containing the `terraform plan` output and verify the planned changes
* Request that the infrastructure team review and merge the pull request

### Running Terraform Locally

You can run Terraform locally with an AWS user having admin permissions.

As the `terraform` and `provider` blocks are generated by Github Actions, we will need to do that first locally as well.
For this reason, we provide the following scripts wrapping around the initialization code:

* `scripts/tfplan` for running `terraform plan`
* `scripts/tfapply` for running `terraform apply`
* `scripts/tfdestroy` for running `terraform destroy`


### Destroy a Stack

After updating `github.env` appropriately, run the script `scripts/tfdestory`.

You can also trigger through GitHub Actions by adding `DESTROY=true` to `github.env`.
GitHub Actions will run `terraform plan -destory` and `terraform destory`, accordingly.
